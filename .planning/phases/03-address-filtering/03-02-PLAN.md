## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| keyboard → React state → render | `addressSearch` is consumed only by `String.prototype.includes` over `address.city` / `address.state` and as React text nodes in the rendered tree. It never crosses the popup→background→content runtime boundary, never reaches Express, and never touches Supabase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | `filteredAddresses` derivation | accept | Pure render-time derivation from in-memory `activeAddresses` (already validated by Phase 2 backend handler). No new IO surface introduced. |
| T-03-05 | Injection | `addressSearch` used in filter / render | mitigate | Value is consumed only via `String.prototype.includes` and rendered only as a React text node / controlled input attribute. No `eval`, no `new RegExp(addressSearch)`, no raw-HTML escape hatches, no DOM string concatenation. |
| T-03-06 | Denial of Service | filter recomputation on each keystroke | accept | Address pool is small (popup-only, single-user scope per CLAUDE.md). Plain `Array.prototype.filter` per keystroke is O(n) over a tiny n; D-08 explicitly defers `useMemo` until profiling shows it matters. |
| T-03-07 | Information Disclosure | filter value at rest | accept | No persistence (D-13); `addressSearch` is discarded on popup close. No logging of the value. |
</threat_model>
